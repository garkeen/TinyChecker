## 表面语法设计

### 类型的表面语法

**定义**（类型宇宙）：`Type`

**定义**（命题宇宙）：`Prop`

Prop 是 Type 的别名，在 Lean 中两者的 universe level 不同，但本项目直接认为两者等同，即两者对应的核心项完全一样。

**定义**（普通函数类型）：普通函数类型用 `->` 连接两个类型，右结合。

例子：`Nat -> Nat`

**定义**（依赖函数类型）：依赖函数类型用 `(IDENT : expr) -> expr` 表示，其中返回类型的表达式可以引用参数名。

例子：`(n:Nat) -> NArray n`

**定义**（和类型）：和类型用 `sum` 关键字定义，列举有限个不相交的构造器。与归纳类型的唯一区别是构造器的参数中不能出现定义自身的类型。

例子：

```
sum Weekday {
| Monday : Weekday
| Tuesday : Weekday
...
};
```

**定义**（积类型）：积类型用 `product` 定义，内部翻译为只有一个构造器的和类型。

例子：

```
product A3d {
  A : Type,
  x : A,
  y : A,
  z : A
};
```

**定义**（归纳类型）：归纳类型用 `inductive` 定义，构造器的参数中可以出现定义自身的类型，支持直接递归和高阶递归。

直接递归示例：

```
inductive NArray (len:Nat) : Type {
| nil : NArray zero
| cons : (x:Nat) -> NArray x -> (new:Nat) -> NArray (succ x)
};
```

高阶递归示例：

```
inductive Tree (A:Type) {
  leaf : A -> Tree A,
  node : (k:Nat) -> (Fin k -> Tree A) -> Tree A
};
```

### 表达式的表面语法

**定义**（等式表达式）：定理证明中一个重要的相等是命题相等，因此给等式类型 `Eq T a b` 提供专用的中缀语法。

例子：`[Nat] zero == zero`

**定义**（Lambda 表达式）：Lambda 表达式用 `\` 开头，参数用 `()` 括起，函数体用 `=>` 引导。多参函数通过 Curry 化实现，即嵌套 Lambda。

例子：`\ (a:Nat) => \ (b:Nat) => add a (add a b)`

**定义**（Let 表达式）：Let 表达式引入一个局部绑定。

例子：`let x:Nat = zero in add x x`

**定义**（Match 表达式）：Match 表达式用于处理归纳类型的值，是归纳子 `D.rec` 的表面语法。

例子：

```
match L as l in List A n bind T k return Nat with
  | nil => zero
  | cons k a xs [ih] => succ ih
  end
```

其中：

- `L` 是被消去的实例值
- `as` 是 motive 中 scrutinee 的名字，可以省略，相当于绑定一个匿名变量
- `in` 后是消去实例的完整类型。必须写成完整的 App 表达式，而且开头必须是归纳类型构造器的名字
- `bind` 是 motive 中为了定义完整归纳类型而需要的参数与索引的变量名字，可以省略，相当于全部变量匿名绑定；不省略的话需要给出全部名字，实在不想给其中的几个起名字，可以用 `_` 显式匿名绑定
- `return` 后面是 motive，即对每个归纳值要计算的返回类型
- 分支中的 `[ih]` 表示归纳假设的绑定名。没有 `[ih]` 时可能是所有归纳假设都匿名绑定，也可能是该分支根本没有递归参数
- 构造器分支必须按定义顺序书写，且必须写全

**定义**（Case 表达式）：Case 表达式用于处理普通和类型的值，是消去子 `D.elim` 的表面语法。与 `match` 的区别在于分支中没有 `[ih]`，因为和类型的构造器不允许递归参数。

### 声明的表面语法

**定义**（变量与断言）：`var` 声明带值的全局变量，`claim` 与之语法相同，区别在于语义分类。

```
var x : Nat = zero;
claim x : [Nat] zero = succ zero -> Empty = ...;
```

逗号分开定义的形式：

```
F : Nat -> Bool -> T,
F n b = E;
```

**定义**（函数与定理）：`fun` 声明计算函数，`theorem` 声明数学定理，两者语法相同。函数体两侧的 `{` `}` 是纯粹的分隔符，不产生 AST 节点。

```
fun add (A:Nat)(B:Nat) : Nat { ... }
theorem mp (A:Type)(B:Type) : (A->B)->A->B { ... }
```

**定义**（公理）：声明无值变量。`axiom LEM : (P:Prop) -> (P -> empty) -> empty -> P;` 随意添加公理会导致系统不一致性。

**定义**（例子）：匿名全局变量。`example : Nat = zero;`

### 形式化语法

#### 词法

```ebnf
IDENT      ::= [a-zA-Z_][a-zA-Z0-9_]*

DOT        ::= IDENT "." IDENT

TYPE_KW    ::= "Type" | "Prop"

KEYWORD    ::= "var" | "fun" | "claim" | "theorem" | "inductive"
             | "sum" | "product" | "axiom" | "example"
             | "let" | "in" | "match" | "case"
             | "as" | "bind" | "return" | "with" | "of" | "end"

MULTI_OP   ::= "=>" | "->" | "=="

SINGLE_OP  ::= "{" | "}" | "(" | ")" | "[" | "]" | "<" | ">"
             | ":" | "=" | "\" | ";" | "," | "|"

COMMENT    ::= "#" [^\n]*
```

重要规则：

- 定义时只能用 IDENT（变量、函数、归纳类型等）
- 普通引用可以用 IDENT 或 DOT（如 `Nat`、`add`、`Nat.rec`、`Bool.elim`）
- 数据构造器不带 `D.` 前缀。构造器声明 `zero : Nat` 会生成全局构造器 `zero`
- 所有全局 IDENT 共享同一个命名空间
- `IDENT.IDENT` 的合法性由繁饰阶段检查

#### 声明语法

```ebnf
decl       ::= var_claim_decl | fun_theorem_decl | inductive_decl
             | sum_decl | product_decl | axiom_decl | example_decl
             | equation_decl

var_claim_decl     ::= ("var" | "claim") IDENT ":" expr "=" expr ";"
fun_theorem_decl   ::= ("fun" | "theorem") IDENT { param } ":" expr "{" expr "}" [ ";" ]
inductive_decl     ::= "inductive" IDENT { param } [ ":" expr ] ctor_block [ ";" ]
sum_decl           ::= "sum" IDENT [ ":" expr ] ctor_block [ ";" ]
product_decl       ::= "product" IDENT field_block [ ";" ]
axiom_decl         ::= "axiom" IDENT ":" expr ";"
example_decl       ::= "example" ":" expr "=" expr [ ";" ]
equation_decl      ::= IDENT ":" expr "," IDENT { IDENT } "=" expr [ ";" ]
```

#### 参数、构造器、字段

```ebnf
param       ::= "(" IDENT ":" expr ")"

ctor_block  ::= "{"  { "|" ctor_decl } "}"
ctor_decl   ::= IDENT ":" expr

field_block ::= "{" [ field_decl { "," field_decl } ] "}"
field_decl  ::= IDENT ":" expr
```

#### 表达式语法

表达式语法设计的核心问题是消除左递归。`expr -> expr`（右结合）和 `expr expr`（左结合）是左递归的，必须消除左递归并划分优先级：

```ebnf
(* 顶层：箭头类型，右结合 *)
expr          ::= expr_no_arrow { "->" expr_no_arrow }

(* 中层：函数应用，左结合 *)
expr_no_arrow ::= expr_no_app { expr_no_app }

(* 底层：所有不含 -> 和应用的表达式形式 *)
expr_no_app   ::= letexpr
                | depiexpr
                | lambdaexpr
                | matchexpr
                | caseexpr
                | parenexpr
                | atomexpr
                | productexpr
                | eqexpr

letexpr       ::= "let" IDENT ":" expr "=" expr "in" expr
depiexpr      ::= "(" IDENT ":" expr ")" "->" expr
lambdaexpr    ::= "\" param "=>" expr
matchexpr     ::= "match" expr [ "as" scrutinee_name ]
                  "in" expr
                  [ "bind" bind_name { bind_name } ]
                  "return" expr
                  "with" { match_branch } "end"
caseexpr      ::= "case" expr [ "as" bind_name ]
                  "in" expr
                  [ "bind" { bind_name } ]
                  "return" expr
                  "of" { case_branch } "end"
parenexpr     ::= "(" expr ")"
atomexpr      ::= IDENT | DOT | TYPE_KW
productexpr   ::= IDENT "<" [ expr { "," expr } ] ">"
eqexpr        ::= "[" expr "]" expr "==" expr
```

#### 模式匹配分支

```ebnf
match_branch  ::= "|" IDENT { field_name } [ "[" ih_name { ih_name } "]" ] "=>" expr
case_branch   ::= "|" IDENT { field_name } "=>" expr

bind_name          ::= IDENT
field_name         ::= IDENT
ih_name            ::= IDENT
scrutinee_name     ::= IDENT
```

### 设计限制

**不实现类型推断**：

- 变量定义，Lambda 表达式参数，Pi 表达式参数都必须写显式类型。可以看到所有的 Decl 中都需要给定义的 IDENT 显式标注声明，Lambda 表达式的参数和 Pi 表达式参数同理。
- 无法实现隐式参数绑定，必须给全参数。无法写出这样的函数 `f:{A:Type}->Eq A x y->Eq A y x`。在 Lean 中，`f` 可以只接收一个类型为 `Eq A x y` 的参数，但是本项目必须接收 `A` 再接受 `Eq A x y`。
- 模式匹配必须写匹配类型和返回类型。在 Lean 中，我们不会标注模式匹配是对什么归纳类型，返回值的类型又是什么。但是本项目为了翻译方便，需要用户手动给出。
- 无法推断宇宙层（当前项目没有宇宙层）。宇宙层的实现过于复杂，本项目简化了这个设计。

**不实现 hole 与 unification**：当前没有 hole 的显式声明，不会根据上下文收集约束并求解。

**不实现 prelude**：没有预置 `Unit`、`Empty`、`Eq`、`refl` 或任何内建消去子。这意味着即使表面语法提供了 Eq 的语法糖，用户依然需要自己定义。

**不实现高级模式匹配**：不做隐式分支推断，不做高级模式匹配细化类型。假设我们有前提 `m : Or (le a b) (le b a)` 并且没有其他信息，那么我们是无法通过消去子得到其中一个 le 的实例，甚至根本不知道 Or 装的是 le 的实例，因为消去子只会获取最顶层的信息。

换句话说，给 `m` 做模式匹配，看到的是一个抽象的 `Or _ _` 类型，而不是具体的 `Or (le _ _) (le _ _)` 类型。因此消去 `m`，只会以 `m:Or x y` 看待，无法细化 `x` 为 `Le a b`，`y` 为 `le b a`。

为了达成从具体类型 `Or (le a b) (le b a)` 中抽取一个 le 的实例的效果，只能通过编码新类型 `leOr a b`，其构造子是 `a->b->(Or (le a b) (le b a))->leOr a b`，这样才能完整的使用这些信息。此时虽然构造子的 `(Or (le a b) (le b a))` 参数看起来没什么用但必须存在，作为一个证明。

**匿名变量 `_`**：`_` 不会注册到上下文，只有绑定位置允许使用。除此之外的任何位置使用 `_` 在语法分析阶段直接拒绝。注意，`_` 依然会占据 de Bruijn index 的一个位置。

**不实现类型宇宙**：当前不实现 Type 的 level 参数，会导致系统不一致，引发 Girard 悖论，之后我们会测试这一点。

### 表面语法的约束

#### 归纳类型与和类型的约束

归纳类型的表面形式为：

```
inductive D (p1:P1) ... (pk:Pk) : I1 -> ... -> Im -> Type {
  c1 : (f1:F1) -> ... -> D p1 ... pk i1result ... imresult
  ...
}
```

**定义**（类型构造器和数据构造器）：

- `D` 被称为该归纳类型的类型构造器
- `c1` 被称为该归纳类型的数据构造器

**定义**（归纳类型的参数、索引和家族）：

- `params = (p1:P1) ... (pk:Pk)`：是归纳类型的参数，在整个定义中保持不变
- `indices = (I1) -> ... -> (Im) -> Type`：是归纳类型的索引
- `family = params + indices`：是归纳类型的家族

**定义**（数据构造器的字段和索引表达式）：

- `fields`：是数据构造器的字段
- `index_results`：构造器返回类型中的索引值（单独看，就是依赖前面 Pi 头变量的索引表达式）

类型构造器的全局类型为：`D : (p1:P1) -> ... -> (pk:Pk) -> (I1) -> ... -> (Im) -> Type`

类型构造器约束：`inductive` 和 `sum` 冒号后面的 `expr` 必须是连续 Pi，最终归宿必须是 `Type`，否则无法解析为 indices 的格式。

数据构造器补齐 params 前缀后的全局类型为：`ci : (p1:P1) -> ... -> (pk:Pk) -> (fields...) -> D p1 ... pk i1result ... imresult`

数据构造器约束：ctor_decl 的 `expr` 必须是连续 Pi，最终归宿是 `APPEXPR`。满足：最终归宿 APPEXPR 被称为目标类型。目标类型头必须是 `D`，目标 params 必须逐项使用原参数名 `p1 ... pk`，目标 family 实参数量必须与定义一致（`k + m` 个）

#### 字段分类

构造器的每个字段按类型中是否出现 `D` 及其出现方式分为三类。

**定义**（直接递归字段）：类型直接以 `D` 为头，如 `succ : Nat -> Nat` 中的 `Nat`。

**定义**（高阶递归字段）：类型是一个函数，最终归宿以 `D` 为头，如 `node` 构造器中的 `Fin k -> Tree A`。

**定义**（普通字段）：除以上两种以外的所有字段。只有直接递归和高阶递归字段生成归纳假设。

递归字段约束：直接递归出现和高阶递归出现的 params 必须逐项使用原参数名 `p1 ... pk`，family 实参数量必须与定义一致（`k + m` 个）

#### 和类型与积类型的约束

和类型约束：

1. 构造器字段不允许出现递归字段
2. 没有参数部分（`param_count = 0`），因为参数本来就是限制递归出现的

积类型约束：

1. 字段名不可重复
2. 字段名不能是 `mk` 或 `elim`
3. 字段类型可以依赖前面的字段变量

#### match 表达式与 case 表达式的约束

`match e as q in D a1 ... aN bind b1 ... bN return R with ...` 中，繁饰时约束：

1. `in` 后面的 expr 作为 APPEXPR 展开，最内层函数 D 必须是全局变量，该全局变量必须是已定义的归纳类型（match）或和类型（case）
2. 参数个数 `a1...aN` 必须等于该类型的 `param_count + index_count`，前 `param_count` 个是 params，后续是 indices

#### DOT 的约束

`IDENT.IDENT`（DOT）在繁饰阶段校验合法性，fst 是第一个 IDENT，snd 第二个 IDENT。DOT 满足下面的约束：

1. `fst` 是归纳类型或和类型，`snd` 是 `rec` 或 `elim`
2. `fst` 是积类型，`snd` 是 `elim`、`mk` 或字段名

除此之外的 DOT 报错。DOT 只能引用已定义的全局名。
