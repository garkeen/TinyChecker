## 类型检查的设计

在此之前，我们需要理解各种相等，以及范式，然后才能知道为什么类型检查需要定义相等。

### 相等的强度层次

**定义**（语法相等）：即两者在 AST 上的结构必须完全一致。`\(f:Nat->Nat) => \(x:Nat) => f x` 和 `\(g:Nat->Nat) => \(n:Nat) => g n` 是语法相等的，`\(x:Nat) => add x 1` 和 `\(x:Nat) => add 1 x` 不是语法相等的。这里默认了 alpha 等价，即两个项可以通过变量重命名互相得到。

**定义**（定义相等）：本项目的定义相等是 alpha 等价加上 Beta/Delta/Iota 等价，不包含 Eta 等价。`\(x:Nat) => 1 + 2 + x` 和 `\(x:Nat) => 3 + x` 是定义相等，`add 3 2` 和 `5` 是定义相等。`\(x:Nat) => add x 1` 和 `\(x:Nat) => add 1 x` 不是定义相等的。

**定义**（命题相等）：即可以通过 Eq 类型证明的相等。如 `\(x:Nat) => add x 1` 和 `\(x:Nat) => add 1 x` 是命题相等的，可以通过归纳法来证明。注意，命题相等不包括 Eta 等价，可以证明 `F x == G x` 对于任意值 `x` 都成立，但无法证明 `F == G`。

**定义**（外延相等）：

$$\frac{\forall x.\, F\ x == G\ x}{F == G}$$

即如果 $F\ x == G\ x$ 对于任意值 $x$，那么 $F == G$。

**定义**（逻辑相等）：

$$\frac{P \to Q \quad Q \to P}{P == Q}$$

即如果 $P \to Q$ 且 $Q \to P$，那么 $P == Q$。

### 相等强度的层次关系

上述五种相等构成了从强到弱的层次：语法相等 $\subset$ 定义相等 $\subset$ 命题相等 $\subset$ 外延相等 $\subset$ 逻辑相等。

考虑这样一个函数：

```
F = \(x:Nat) => match x in Nat return Type with
     | zero => Bool
     | succ x => Bool
     end
```

可以知道，无论 x 是什么，我们都有 `F x = Bool`。那么 `\(x:Nat) => \(y:F x) => expr` 和 `\(x:Nat) => \(y:Bool) => expr` 行为一样，但是它们定义相等吗？答案出乎意料：不是。即便两个函数对同一输入立即定义相等，仍不成立——因为这个相等无法通过规约得到。由于 x 没有任何构造器头，我们不会触发 iota 规约。但是 `F` 和 `\(x:Nat) => F x` 确实是定义相等的，因为 `F` 有函数体，因此 `F x` 可以进行 Beta 规约，结果与 `F` 的函数体完全一致，且两者的 Lambda 头也一致，所以 `\(x:Nat) => F x` 与 `F` 定义相等。

### 范式

**定义**（范式 NF）：NF 指的是，不存在任何一个可以规约的子结构。可以规约的子结构叫做 redex，因此 NF 相当于表达式中不存在 redex。

任何项都可以看成一棵树，那么不存在任何一个可以规约的子结构，相当于不存在一个子节点可以继续执行规约动作。

**定义**（弱头范式 WHNF）：指的是头部不可再规约的表达式。

- 对于 Lambda 表达式和 Pi 表达式，它们就是 WHNF，其头部可以看成就是 `=>` 和 `->`。
- 对于单个变量，它是 WHNF 如果这个变量不可再进行 delta 规约。
- 对于一个 app 表达式，它是 WHNF 如果它的头部是无值变量无法再展开。当然，消去子是例外，需要判断当前 app 的参数无法满足 iota 规约。

在判断相等的时候，如果两个项确实相等，那么 WHNF 相比于 NF，不会减少任何计算量，该算的还是要算。WHNF 真正的优势是早点失败，避免冗余计算，即先比较头部，再递归比较参数的头部，一旦不相等即停止，比如：

```
add 100 3 和 add 3 1
```

是不同的。如果使用 NF，则会一步步地将：

```
add 100 3 → succ (add 99 3) → succ (succ (add 98 3)) → ...
add 3 1 → succ (add 2 1) → succ (succ (add 1 1)) → ...
```

一直规约到完全展开。而如果使用 WHNF，左侧展开几步后在 `succ^5 (add 95 3)` 处停止，右侧展开后在 `succ^4 zero` 处停止，因为两者头部 `succ` 的层数不同，直接判定不相等，避免了后续无意义的计算。

**定义**（合流性）：事实上，对于同一个项，我们规约的第一步可以出现在不同的地方。比如 $(\lambda x.\, 1 + 2 + x)\ (3 + 4)$ 中，我们第一步可以把 $1+2$ 规约为 $3$，也可以把 $3+4$ 规约为 $7$。那么，能否保证，即使规约路径不同，NF 也是相同的？Church-Rosser 定理提供了上述问题的保证。这个定理指出：Lambda 演算的规约具有合流性，即若 $A \twoheadrightarrow B$ 且 $A \twoheadrightarrow C$，则存在 $D$ 使得 $B \twoheadrightarrow D$ 且 $C \twoheadrightarrow D$。因此可以得出一个推论：相同的表达式无论选择什么样的规约路径，只要达到了不可规约的 NF 状态，那么这些 NF 一定相同。

因此，我们可以选择一直从头部规约（head first）的算法，这被称为正常序。而理论上，我们可以先算出参数的范式，再对头部进行规约，这被称为应用序。

值得一提的是，在命令式语言如 C 语言中，调用函数本质上是先算出参数的具体值，再进行代入。

### 设计理念

所有编程语言的类型检查都包含两种操作：

- **推导**一个项的类型
- **检查** A 类型是否等于 B 类型

比如最简单的 `int x = 1;` 推导出 1 的类型为 int，检查 int 是否等于 int。这一点任何编程语言都符合。在简单类型编程语言中，对于任意声明，我们都可以直接推导左侧的类型，然后与右侧类型直接进行比较。但是在依赖类型编程语言，就有问题了：类型依赖于项，我们在推导的时候会做 Beta 规约，把之前输入的值带入之后类型注解的项上，其次，我们不仅需要比较类型的相等，还需要比较值的相等。

比如，`List Nat 3` 和 `List Nat (1+2)` 比较后是相等的吗？我们不能直接进行比较，因为值的相等不是直接语法相等，而是定义相等，需要通过所谓的转换检查规约到 NF 再判定。

### 双向类型检查

所谓的双向类型检查（bidirectional type checking），指的是，推导类型的时候会发生检查类型，检查类型的时候又会出现推导类型。虽然整体来说，本项目不支持任何在 Pi 头或者 Lambda 头的类型注释省略，或者隐式参数绑定，执行单向的类型检查就足够了。但我们依然使用双向类型检查来实现本项目对象语言的类型检查功能，这是为了之后如果项目需要支持隐式参数绑定或者类型注释省略的扩展性可能做支持。

之前我们说过，类型检查分为推导和检查。这个项目也是如此：我们区分为推导模式和检查模式。

**推导模式**：给定一个项，根据局部和全局上下文，推导出它的类型。由于过程中会出现规约，而规约需要保证类型正确，因此会调用检查模式。

**检查模式**：给定两个类型，通过**转换检查**判定是否相等，无需调用推导模式。

推导的逻辑在实现的类型检查章节中已经阐述，这里不再赘述了。我们的重点在于如何实现转换检查。

### 转换检查

转换检查（conversion check）是判断两个项是否定义相等的过程。由于依赖类型的存在，推导出的类型和注解类型可能在语法上不同但定义相等，例如 $\text{add}\ \bar{1}\ \bar{2}$ 和 $\bar{3}$。因此，类型合一必须使用定义相等而非语法相等。

转换检查在类型检查中的使用时机是：每当推导出一个类型 $T_{\text{inferred}}$，而注解给出的类型是 $T_{\text{annotated}}$ 时，调用 $\text{convert}(T_{\text{inferred}}, T_{\text{annotated}})$。若成功则继续，若失败则报类型错误。

下面以设计演进的顺序说明转换检查算法。

#### 直接 NF 判等

假设我们需要证明存在项：

```
ADDEQ : [Nat] add(200,300) == add(300,200)
```

我们当然可以直接写：

```
var x : ADDEQ = refl Nat add(200,300)
```

就会触发：

```
EXPR1: [Nat] add(200,300) == add(300,200)
EXPR2: [Nat] add(200,300) == add(200,300)
```

的判等。这时如果我们选择直接 NF 的转换检查，会把 `add(200,300)` 和 `add(300,200)` 完全展开为 succ 的复合项，大概类似于：

```
EXPR1: [Nat] succ succ ... zero == succ succ ...
EXPR2: [Nat] succ succ ... zero == succ succ ...
```

缺点十分明显，证明项十分巨大。

另一种情况，假设我们已经证明了：

```
Addswap : (m:Nat) -> (n:Nat) -> [Nat] add(m,n) == add(n,m)
```

那么 `Addswap 200 300` 这个项 infer 的结果是：

```
Addswap 200 300 : add(200,300) == add(300,200)
```

如果我们写：

```
var x : ADDEQ = Addswap 200 300
```

那么就会触发：

```
EXPR3: [Nat] add(200,300) == add(300,200)
EXPR4: [Nat] add(200,300) == add(300,200)
```

的判等，选择直接 NF 的转换检查，就会直接展开 `refl Nat add(200,300)` 再判断相等。但是因为我们知道，如果函数一样，参数一样，那么不用算也知道值一定一样，而 EXPR3 和 EXPR4 的 AST 完全相同，因此不需要展开为 NF 就知道一定相等。假设算法不管三七二十一，无论判断什么都直接求完全范式再判等，那么还是会把 `add(200,300)` 完全展开，还是会变成：

```
EXPR3: [Nat] succ succ ... == succ succ ...
EXPR4: [Nat] succ succ ... == succ succ ...
```

产生庞大的证明项。因此，我们的目标是，求值器必须不是无脑的判定相等，也就是无论如何都先求个范式再说，而是应该充分利用中间态的语法相等简化计算。

#### 语法相等短路，不等再求 NF

第一个方法是直接判定语法相等，相等就直接返回定义相等，不等就规约 NF 再判等。我们的底层有 CApp、CLam、CPi、CVar、CGlobal、CType。其中两个 CApp 判断最复杂，因此下面我们以两个 CApp 为例子。

```
EXPR1: A B
EXPR2: C D
```

转换检查的时候，不要先执行把 B 带入 A 的规约，而是比较 A 和 C 是否直接语法相等，B 和 D 是否直接语法相等。如果头相等，参数相等，那么一定是定义相等，无论头有没有值都无需求值。如果头不相等，而且两侧头没有值，则无需求到 NF，直接判断不可能相等。其他情况下，都需要求到 NF。

简单的来说，先看两个表达式是不是同一种（都是 CLam，都是 CPi），是的话判断是否语法相等。不是的话看头是不是不一样，比如 CLam 和 CPi 不可能相等，直接结束。如果无法判断，比如 CApp 和 CLam，就只能求 NF 了。

#### 避免 Beta 规约的 NF 判等

直接判定语法相等，不等就规约到 NF 代价依然很大。继续以 CApp 为例子，比如：

```
EXPR1: add 3 4
EXPR2: add 3 (add 2 2)
```

我们知道 4 和 `add 2 2` 不是语法相等，但是如果把 EXPR2 的 `add 2 2` 规约一下，就会变成：

```
EXPR1: add 3 4
EXPR2: add 3 4
```

是语法相等，结束。因此提出第二个办法，比较 A B 和 C D，先比较是否语法相等；不是的话递归比较 A 和 C 是否定义相等，再比较 B 和 D 是否定义相等，是的话直接返回对，不是的话 B 带入 A，D 带入 C，之后继续判断。当然，如果两个表达式不是同一种（比如 CApp 和 CLam），就只能求 NF 了。

为什么 A 和 C 不是定义相等不直接返回失败？一个最简单的反例是：

```
EXPR3: add 3 4
EXPR4: min 10 3
```

它们的头绝对不相等，但是它们的值都是 7。

上面的方法依然不完美。考虑下面的例子：

```
Γ => F : Nat -> Nat = add (add 1 2)
EXPR1: add 3 4
EXPR2: F 4
```

两者的头分别是 `add 3` 和 `F`，肯定不等，那么接下来就是求 `add 3` 和 `F` 到 NF 吗？如果我们把 F 带入，那么 `F 4` 变成 `add (add 1 2) 4`，它的头不再是 F 而是 `add (add 1 2)`。此时有：

```
EXPR2: add (add 1 2) 4
```

与 EXPR1 的头 `add 3` 仍然不相等，继续展开 `add (add 1 2)` 才能找到匹配。因此，第三个办法一个可能的思路是，先比较 A 和 C 是否语法相等，如果不相等，那么尝试规约 C 一次，再判断。如果最终两者不等当然只能大规约。若两者相等，问题则变成了判断参数 B 和 D 是否语法相等。如果不是，尝试规约 D 一次，再判断。如果参数规约到 NF 还不相等，那么就只能执行大规约。

到这里，麻烦来了，A 和 C 没有区别，对于上面的例子，EXPR1 和 EXPR2 互换，把 C 规约到 NF 都无法得到头语法相等，但是规约 A 就能一次出现语法相等。那么，存在一个简单的规约策略，使得经过最少的左右规约选择，就能得到语法相等呢？回顾我们上面的操作，我们是知道 F 展开一次后恰好会头部相等，但是解释器不知道。对于解释器来说 F 和 `add 3` 都是不完整的表达式，没有区别，如何知道下一步是展开左侧而不是右侧？而且，例子设计的恰好展开一次后头部就相等，现实情况可能需要规约多次。解释器必须判断，规约一次后不相等，接下来继续展开左侧还是右侧？

这就说明一个问题，为了尽快的利用语法相等，也就是尽量让两个项的头相同（头部不一定变成无法继续规约的状态，比如 add 有定义，其实是可以继续规约的），然后比较参数，解释器必须聪明的选择对左侧规约一次还是右侧规约一次。然而，除非你已经有了头部的完整规约到 NF 的路径图，否则不可能知道最优规约路径是什么，不过知道了规约到 NF 的路径图，再退回来选取一个路径规约没有任何意义，都可以直接比较 NF。所以，第三条路似乎根本无法设计出来。

好消息是，我们不需要最优规约路径，而是**局部最优规约路径**，即使用贪心策略。虽然对于解释器来说 F 和 `add 3` 都是不完整的表达式，但是左侧是 CGlobal，右侧是 CApp，直觉上展开 CGlobal 是更好的选择。上述的例子恰好佐证了这一点，我们展开 F，便可以得到相同的头部了。至于规约一次后还是不相同，继续展开左侧还是右侧，可以变成一种递归判断，因为两侧表达式的比较是可以通过表达式类型来分类的，我们只要编码每种情况的状态需要执行的动作，然后看下一个状态是什么就行了。

第三种尝试需要把 CVar/CGlobal 分成有值 CGlobal 和无值：有值 CGlobal 可以 Delta 展开。无值包括 CVar、类型构造器、数据构造器、裸消去子。CApp 需要优先判断整体 application 是否可以规约，包括 Beta、Delta、Iota，以及消去子 scrutinee 为了触发 Iota 而进行的规约。

总之，先判断语法相等，再尝试判断左右侧是否定义相等。如果必须规约，选择 Delta 规约比 Beta 规约更好。

#### 第三种办法的缺陷

现如今我们有完善的第三种办法，但是某些时候，提前计算参数反而会弄巧成拙，比如：

```
EXPR1: F (add 2 2) (pow 3 3)
EXPR2: F 4 (sub 30 3)
```

如果发现 F 头一样，然后算 `add 2 2` 和 4 也一样，然后计算 `pow 3 3` 和 `sub 30 2` 发现不相等，之后才会发生 F 的应用。但是如果 F 非常简单，就像 `F = \x => \y => zero`，那么比较参数其实是没有用的，因为参数根本不会加入到 F 的函数体，这个时候计算参数反而变成了累赘。这意味着，有些时候，比较 EXPR1: A B 与 EXPR2: C D 先进行大规约 (A B) (C D) 而不是小规约 A B C D 会更好。因此，解释器不仅需要决定先规约左侧还是先规约右侧，规约多少次，还需要决定到底是先小规约 A B C D 还是先大规约 (A B) (C D)。

#### WHNF 判等

使用 WHNF 的方式而不是 NF 的方式判等，并吸纳前面思考的一个想法，在中间过程加入快速判断。由于 WHNF 是一直规约到头部不可规约，我们不需要思考先规约左侧还是右侧，先小规约还是大规约。这就是本项目的实现方式。

#### 避免 Delta 规约的贪心 WHNF 判等

在第三种尝试的基础上，避免 Delta 规约，贪心的进行 Beta 和 Iota 规约。整体流程是一个循环：

1. 快速语法判等——成功则直接返回。
2. 在只做 Beta 与 Iota 规约（不做任何 Delta 展开）的前提下求 WHNF 判等——成功则返回。
3. 如果以上两步都失败，必须在所有可展开的 CGlobal 中选择一个进行 Delta 展开，展开后回到步骤 1。
4. 如果没有任何 CGlobal 可以展开（即两侧的所有 CGlobal 都是无值变量），判定失败，两边不相等。

这里最妙的地方在于步骤 3 的选择策略。简单的来说，不是随机挑选一个 CGlobal 展开，而是提前计算一个全局变量的**高度**。高度定义为：有值的 CGlobal，它从底层的构造器/公理开始算，作为叶子节点在引用树上的高度。具体规则：

- 构造器、公理等无值定义：高度为 0。
- 有值定义 `x = ...`：高度为 $1 + \max(\text{高度}(a_1), \text{高度}(a_2), \ldots)$，其中 $a_i$ 为该定义体引用的所有全局名。

直觉上，高度越低，越接近底层构造器，展开后越容易暴露结构上的相似性，因此每次应该选择当前两边所有有值 CGlobal 中高度最低的那一个进行 Delta 展开。

高度是静态的——类型检查开始前一次性预计算全体全局定义即可，不会随着 Delta 展开而改变。这恰好解决了第三种尝试中"先规约左侧还是右侧""先小规约还是大规约"的选择困境：Beta/Iota 随意做，一旦必须要 Delta 了，就挑最浅的展开。事实上，Lean 内核的转换检查采用的就是类似思路。

### 消去子类型检查的两种机制

以下面的定理为例：

```
theorem eqSym (A:Type) (x:A) (y:A) (p:[A]x==y): [A]y==x {
  match p as e in Eq A x y bind T rhs lhs
  return [T]lhs==rhs with
  | refl z => refl T z
  end
}
```

- **实例化消去子**：在编写 branch 阶段，归纳类型的参数 T rhs lhs 是通过构造子的参数变量依赖得到的。从签名就可以看出这一点。此时，目标类型 `[T]lhs==rhs` 中的各个归纳类型参数 T rhs lhs 会被填入该分支的计算结果。
- **使用实例化的消去子**：在消去 scrutinee 阶段，目标归纳类型的参数并不是当前 scrutinee 通过 infer 得到的，而是当前 scrutinee 的注释信息 A x y，这样我们才能证明出 `Eq A y x`。如果不这样，在 match 消去 p 值的时候，使用 p 通过 infer 得到的参数，那么由于 p 通过 infer 得到的类型只能是 `Eq A m m` 或者 `Eq A x x`（即无论如何后面两个值一定一样），模式匹配的结果只能是 `Eq A m m` 或者 `Eq A x x`，而不是 `Eq A y x`。
