#1
(\x.\y.\z.p) x y z m n
能否在elab阶段就避免所有的参数个数不匹配问题 不能
因为p可以是消去子的表达式 根据规则可以是返回一个2个输入的lambda表达式 或者3个输入的lambda表达式
这样只有计算完p才知道能不能完整的输入m和n
如果p是2输入 那么这个是合法的参数个数匹配
如果p是3输入 那么这不合法
除非你求值了 求值不属于elab
比如
var P:Nat->Type = (
    match x in Nat return Type with 
    |zero => Nat 
    |succ x => Nat->Nat 
    end)
var F1:(x:Nat)->P x = \(x:Nat) =>
    match x in Nat return P x with
    |zero => zero 
    |succ x => \(y:Nat) => add x y 
    end
var t1:Nat = F 0
var t2:Nat = F 1 1
t1和t2 都是合法的

如果我们写成
var F2：(x:Nat)->P x->Nat
那么F的行为不会是我们一开始想象的样子，可以写出
var t1:Nat = F 0 0
var t2:Nat = F 1 1 1
为什么呢？因为
(match x in Nat return Type with |zero => Nat |succ x => Nat->Nat end)->Bool
与
(match x in Nat return Type with |zero => Nat->Bool |succ x => Nat->Bool end)
不是同一个东西 在match内的任何->都低于外面的->一个级别 而不是同级的
因此P x在x等于1的时候会识别为一个函数类型

事实上，如果你想要P x里面的箭头和外面一个级别，就无法在语法分析的时候就知道F2的语法树了，因为你无法建立一个AST，对应类型是
Nat-> if x=0 then Nat else Nat->Nat ->Bool
注意，这里Nat->Nat的->与外面同级别，也就是此时输入一个x如果是1那么还需要两个参数才能输出true
这种操作意味着我们允许修改AST的中间节点而保持子节点不变，在这个项目这是不可能的，任何操作都是对一个完整的子树修改

一个正确的程序 不管类型检查是否能通过 不管能不能停机 他首先就有一个要求 那就是必须是闭合的
也就是说 任何被使用的变量都必须被定义 必须已经在上下文存在
要么通过global注册了类型和值 要么通过pi->和lambda=>注册了类型 但是暂时不知道值
因此有两个上下文 全局上下文global和局部上下文local
一个变量不可能自由出现 自由出现就是错误的程序
global上下文的定义是有顺序的 不允许覆盖的
local上下文的定义也是有顺序的 允许覆盖的（shadow）
在local中 由于我们的等价允许alpha等价 所以局部上下文需要debruijn index 全局定义不需要
因此如果出现值的别名 比如var x:Type= Nat ;var y:x=zero
还需要beta等价才能确定
这个项目的规约只有beta规约和iota规约

#16
这个语言无法实现特设多态，因为每一个函数头类型在当前的上下文拥有确定唯一的NF(考虑到函数头类型可能包裹在一个函数体内，那么其依赖的变量依然可能没有值，但是没问题，因为参数和函数必须在同一个上下文)，因此，假设存在一个函数实现了特设多态
F： EXPR -> Unit
其中，F zero和F false 都合法，由于EXPR具有唯一的NF，那么可以推导出
EXPR == Nat 而且 EXPR == Bool
于是Nat == Bool 这是不可能的
注意，我们当然可以写出
F: {T:Type} -> T -> Unit
在支持隐式参数绑定和元变量求解的情况下，F zero和F false 都合法，看起来我们之前错了，其实没错，因为这里的函数头不是T，而是{T:Type},具有类型Type是唯一的NF，不是例外
