## 词法分析和语法分析

本系统的词法分析器与语法分析器均采用手写实现，不依赖 Lex/Yacc、ANTLR 等外部生成工具。这样做的原因在于，本项目的表面语法规模较小，且第三章给出的文法已经消除了左递归，适合使用递归下降方法直接实现。

词法分析阶段的输入是源码字符串，输出是 token 序列：

$$
\text{String} \to \text{List}[\text{Token}]
$$

语法分析阶段的输入是 token 序列，输出是表面语法声明列表：

$$
\text{List}[\text{Token}] \to \text{List}[\text{Decl}]
$$

其中 `Decl` 与 `Expr` 均为表面语法树节点，后续交由繁饰阶段继续处理。

### 词法分析

词法分析器以单次线性扫描的方式读取源码。扫描过程中维护当前位置下标、行号、列号，并在识别每个 token 时记录其原始文本和源码区间。一个 token 至少包含以下信息：

1. `kind`：记号类别，如 `IDENT`、`FUN`、`ARROW`；
2. `text`：源码中的原始文本；
3. `row` 与 `col`：token 开始处的行列位置；
4. `start_pos` 与 `end_pos`：token 在源字符串中的字符区间。

其中行列位置主要用于错误报告，字符区间用于在需要时把错误信息定位回源码片段。

本系统识别的 token 可以分为五类。

1. **标识符与点号引用**。普通标识符由字母或下划线开头，后续可包含字母、数字与下划线。若标识符后紧跟一个点号和另一个标识符，则整体识别为 `DOT`，如 `Nat.rec`、`Pair.x`。
2. **关键字**。声明关键字包括 `var`、`fun`、`claim`、`theorem`、`inductive`、`sum`、`product`、`axiom`、`example` 等；表达式关键字包括 `let`、`in`、`match`、`case`、`as`、`bind`、`return`、`with`、`of`、`end` 等。
3. **类型关键字**。`Type` 与 `Prop` 被单独识别为类型关键字。二者在繁饰阶段都会被翻译为同一个核心项 `CType`。
4. **多字符运算符**。包括 `->`、`=>`、`==`，分别用于函数类型、Lambda 表达式和等式语法糖。
5. **单字符分隔符**。包括花括号、小括号、方括号、尖括号、冒号、等号、反斜杠、分号、逗号和竖线。

注释以 `#` 开始，持续到当前行末尾。词法分析器会跳过空白符和注释，但仍会正确更新行列信息。

点号引用也在词法阶段做最基本的形态限制：`IDENT.IDENT` 是合法形式，但不允许连续出现多个点号，也不允许点号后缺少标识符。至于 `Nat.rec` 或 `Pair.x` 是否真的是合法引用，则不在词法阶段判断，而是推迟到繁饰阶段根据全局上下文和类型构造器信息检查。

### 语法分析

语法分析器采用递归下降方法实现。分析器维护一个 token 指针，每个语法函数负责识别一种语法结构，并在成功后推进指针；若当前 token 不符合预期，则抛出带行列信息的语法错误。

由于表面文法已经按照优先级分层，且每种声明和表达式形式都有稳定的起始 token，因此语法分析过程可以看作一个 LL(2) 分析过程：大多数分支只需查看当前 token 即可决定，少数位置需要多看一个 token。例如遇到 `(` 时，若后续形如 `IDENT COLON`，则进入依赖函数类型或参数解析；否则按普通括号表达式处理。遇到 `IDENT` 时，若后续是 `<`，则解析为积类型构造表达式；否则解析为普通原子表达式。

实现中还显式传递终止符集合 `stop`。也就是说，每个表达式解析函数都知道自己应当在哪些 token 前停止。由于终止符在每个语法位置都是确定的，递归下降过程不会在长表达式中无界吞入后续结构，也不需要回溯。

顶层入口反复调用声明分析函数，直到遇到 `EOF`。声明的解析分支由起始 token 唯一决定：

| 声明形式 | 起始 token | 内部表达式的主要终止符 | 结果节点 |
| --- | --- | --- | --- |
| `var` / `claim` | `VAR` / `CLAIM` | 类型到 `EQ`，值到 `SEMI` 或 `EOF` | `VarDecl` |
| `fun` / `theorem` | `FUN` / `THEOREM` | 返回类型到 `LBRACE`，函数体到 `RBRACE` | `FunDecl` |
| `inductive` / `sum` | `INDUCTIVE` / `SUM` | arity 到 `LBRACE`，构造器类型到 `BAR` 或 `RBRACE` | `InductiveDecl` |
| `product` | `PRODUCT` | 字段类型到 `COMMA` 或 `RBRACE` | `ProductDecl` |
| `axiom` | `AXIOM` | 类型到 `SEMI` | `AxiomDecl` |
| `example` | `EXAMPLE` | 类型到 `EQ`，值到 `SEMI` 或 `EOF` | `ExampleDecl` |
| 等式定义 | `IDENT` | 类型到 `COMMA`，值到 `SEMI` 或 `EOF` | `EquationDecl` |

表达式解析分为 `parse_expr`、`parse_app` 和 `parse_no_app` 三层，对应第三章中消除左递归后的表达式文法。

| 层级 | 实现函数 | 处理对象 | 结合方向 | 关键逻辑 |
| --- | --- | --- | --- | --- |
| 1 | `parse_expr(stop)` | 箭头类型 `A -> B` | 右结合 | 先调用 `parse_app(stop ∪ {ARROW})` 读左侧；若当前 token 是 `ARROW` 且 `ARROW` 不在 `stop` 中，则递归调用 `parse_expr(stop)` 读取右侧 |
| 2 | `parse_app(stop)` | 函数应用 `f a b` | 左结合 | 先读一个 `parse_no_app(stop)`；当下一个 token 属于表达式起始符且不在 `stop` 中时，继续读取参数并构造左结合 `AppExpr` |
| 3 | `parse_no_app(stop)` | 不再拆分应用的基本表达式 | 无 | 根据当前 token 分派到 `let`、Lambda、`match`、`case`、等式、括号、原子和积类型构造等具体解析函数 |

这三层结构避免了形如 `expr ::= expr expr` 或 `expr ::= expr "->" expr` 的左递归。例如 `A -> B -> C` 会被第一层解析为 `A -> (B -> C)`；`f a b` 会被第二层解析为 `(f a) b`。箭头类型和函数应用的优先级也由函数调用顺序确定：应用先在第二层聚合，随后第一层才检查是否存在顶层箭头。

### 复杂表达式的解析

`let`、等式语法糖、`match` 与 `case` 的具体语法已经在第三章用 EBNF 给出。实现时不再为这些结构设计额外的分析算法，而是直接按照关键字顺序逐项消费 token，并在每个子表达式处传入确定的终止符集合。

表达式形式的起始符与主要终止符如下：

| 表达式形式 | 起始 token | 内部子表达式的主要终止符 | 结果节点 |
| --- | --- | --- | --- |
| 原子表达式 | `IDENT` / `DOT` / `TYPE` | 无 | `AtomExpr` |
| 积类型构造 | `IDENT` 后接 `LANGLE` | 每个参数到 `COMMA` 或 `RANGLE` | `ProductExpr` |
| 普通括号 | `LPAREN` 且后续不是 `IDENT COLON` | 内部表达式到 `RPAREN` | 内部表达式本身 |
| 依赖函数类型 | `LPAREN IDENT COLON` | 参数类型到 `RPAREN`，返回类型继续按表达式解析 | `PiExpr` |
| Lambda | `LAMBDA` | 参数类型到 `RPAREN`，函数体继续按表达式解析 | `LambdaExpr` |
| `let` | `LET` | 类型到 `EQ`，值到 `IN`，body 继续按表达式解析 | `LetExpr` |
| 等式语法糖 | `LBRACKET` | 类型到 `RBRACKET`，左侧到 `EQEQ`，右侧继续按表达式解析 | `EqExpr` |
| `match` | `MATCH` | scrutinee 到 `AS` 或 `IN`，族实例到 `BIND` 或 `RETURN`，motive 到 `WITH`，分支体到 `BAR` 或 `END` | `MatchExpr` |
| `case` | `CASE` | scrutinee 到 `AS` 或 `IN`，族实例到 `BIND` 或 `RETURN`，motive 到 `OF`，分支体到 `BAR` 或 `END` | `CaseExpr` |

对 `match` 与 `case` 中 `in` 后面的类型族实例，语法分析器会先解析为普通应用表达式，再拆出应用头与参数列表。应用头必须是原子表达式，否则无法确定当前表达式试图消去哪一个类型族。分支中的字段名和归纳假设名仍然只是表面名字，其数量、顺序和类型是否正确，留给繁饰阶段结合归纳类型定义进行检查。

因此，复杂表达式在语法阶段只完成结构化，不执行类型层面的合法性判断。例如 `case` 是否真的作用于和类型，`match` 的分支是否与构造器顺序一致，`bind` 数量是否与类型族参数和索引数量一致，都由繁饰阶段处理。

### 语法分析结果

语法分析阶段的主要输出是声明列表 `List[Decl]`。声明节点包括变量声明、函数声明、归纳类型声明、积类型声明、公理声明、例子声明和等式定义；表达式节点包括原子、应用、箭头类型、依赖函数类型、Lambda、`let`、`match`、`case`、积类型构造和等式表达式。

语法分析阶段只负责保证源码满足表面语法，并把线性的 token 序列组织为树形结构。它不会判断名字是否已经定义，也不会判断类型族、构造器顺序、递归字段和点号引用是否合法。这些与上下文和类型信息相关的检查，统一交由繁饰阶段完成。
